# What is a Graph Database? # > The graph paradigm goes well beyond databases and application development; it’s a reimagining of what’s possible around the idea of connections. [Neo4j blog](https://neo4j.com/blog/why-graph-databases-are-the-future/)
# Graphs in Computer Science * Graph (G) is a Vertext (V) and an Edge (E) * Vertext is an object * Edge defines the relationship

Sample Graph

  • John and Clark know each other and are friends. Relationship is two way
  • Carrie and Clark know each other, Carrie is the boss. Relationship is one way

Graph Theory - Vertices

  • Vertices can have a label (Person, Event)
  • Vertices can have arbitrary key-value properties (Birth, Death, Intro)

Graph Theory - Edges

  • Edges defined the relationships between Vertices
  • Edges can have a label
  • Edges may have multiple set of key-value pairs for properties (which are queryable)
  • A relationship could be a friend and a boss

## Graph Databases ## Two Types 1. Resource Definition Framework (RDF) Triplestore (W3C/semantic web) 1. Labled Property Graph
## It Is Complicated ## * Standards don't really exist * Multiple vendors, multiple implementations
## Query Languages ## * SPARQL - W3C Standard query language for the semantic web. Some graph databases support * Cypher (and openCypher) - Query langauage developed by Neo4J and opened source in 2015 * Gremlin - Part of Apache TinkerPop. Implemented widest variety of vendors, including Auzure Cosmos DB, AWS Neptune, Janus Graph, etc. Even Neo4J implements * GraphQL - Some vendors support GraphQL * SQL-like things - A whole bunch of folks have SQL-esque implementations
## Graph Databases ## * SASS - AWS Neptune, Azure Cosmos DB * Commercial - Neo4j, Datastax * OSS - JanusGraph
## Choice ## * TinkerPop/Gremlin over Neo4j/Cypher * Gremlin language is most portable * TinkerPop implementations abound
## Installation Notes on my GitHub(https://github.com/jptacek/that-conf-2019-graph-database) Essentially follow and run
# Querying Creating the graph 

			    g = graph.traversal()
# Querying List all the things 
					g.V()
# Querying Get the first vertex 
					g.V(1)
# Querying Get a property off of the first vertex by key 
					g.V(1).values('name')
					g.V(1).values('age')
# Querying Aggregate together 
					g.V(1).values('name','age')
# Querying List all of the edges out of a vertex with a label property on the edge 
					g.V(1).outE('knows')
# Querying List all of the names. In this case, we need to get the connecting vertex. That vertext is an incoming node to the edge, so we use the inV keyword 
					g.V(1).outE('knows').inV().values('name','age')
# Querying We can also have copmarison operators 
					g.V(1).outE('knows').inV().has('age',gt(30)).values('name','age')
# Querying We are going to start over. Quit command line (:q) and restart 
					:q
# Querying Create conection 
					graph = TinkerGraph.open()
# Querying Create graph traversal 
					g = graph.traversal()
# Querying Add a vertex 
					v1 = g.addV('person').property(id,1).property('name','marko').property('age',27)
# Querying Add person the second 
					v2 = g.addV('person').property(id,2).property('name','josh').property('age',32)
# Querying Create an edge. This will connnect two item and provide a type of relationship 
					g.V(1).addE('knows').to(g.V(2))
